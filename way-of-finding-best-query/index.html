여정 of 분산 쿼리

September 25, 2020

안된다.
이 시대에 분산 쿼리라니
요즘 세상에 linked server라니

모델은 간단하다.
서버 a에서 서버 b의 테이블을 조인으로 연결해 조회하면 된다.
살짝 역겨운 것은 10개의 테이블을 UNION ALL한 뷰가 본 서버의 대상 테이블이다

 

  1. 일반 outer join
SELECT a.*, i.*
FROM a_tbl a
LEFT OUTER JOIN OPENQUERY(i, 'SELECT * FROM db.dbo.tbl') i ON a.id = i.id
ORDER BY a.date DESC
OFFSET 1 ROWS FETCH NEXT 100 ROWS ONLY;

놀랍지 않겠지만 반올림한 50초의 실행시간이 걸린다.

 

  1. 임시 테이블
WITH i AS (SELECT * FROM OPENQUERY(i, 'SELECT id FROM db.dbo.tbl'))
SELECT *
FROM a_tbl a
LEFT OUTER JOIN i ON a.id = i.id
ORDER BY a.date DESC
OFFSET 0 ROWS FETCH NEXT 100 ROWS ONLY;

여전히 무난한 40초

 

  1. 사전 정의된 조건절

sql quote의 escape는 두번 사용이다

DECLARE @tmp VARCHAR(5000);
SET @tmp = ''
SELECT @tmp = @tmp + '''''' + id + ''''','
FROM (SELECT id, status FROM a_tbl 
        ORDER BY date DESC
        OFFSET 0 ROWS FETCH NEXT 100 ROWS ONLY) a
WHERE a.status IS NOT NULL  
DECLARE @sql VARCHAR(5000);
SET @sql = 'SELECT *
        FROM a_tbl a
        LEFT OUTER JOIN OPENQUERY(i,  
                ''SELECT id FROM db.dbo.tbl
                WHERE id IN ('+LEFT(@tmp, LEN(@tmp) -1)+')'') i
        ON a.id = i.id
        ORDER BY a.date DESC
        OFFSET 1 ROWS FETCH NEXT 100 ROWS ONLY'
EXEC SP_SQLEXEC @sql;

20초

 

  1. 사전 정의된 조건절 2

변수에 할당된 길이를 다 사용하여 query가 잘렸다.

DECLARE @tmp VARCHAR(7000);
SET @tmp = '';
SELECT @tmp = @tmp + ''''+id+''','
FROM a_tbl a
ORDER BY a.date DESC
OFFSET 0 ROWS FETCH NEXT 100 ROWS ONLY;
SET @tmp = LEFT(@tmp, LEN(@tmp) -1);  
DECLARE @sql VARCHAR(max);
SET @sql = 'SELECT *
        FROM a_tbl a
        LEFT OUTER JOIN OPENQUERY(i, 
                ''SELECT id FROM db.dbo.tbl WHERE id IN ('replace(@tmp,'''','''''')+')'') i 
        ON a.id = i.id 
        WHERE a.id in ('+@tmp+') 
        ORDER BY a.date DESC';
exec sp_sqlexec @sql;

10초 안쪽으로도 나오지만 여전히 느리다
조회 목표값이 커질 수록 변수의 max value의 한계로 인해 실사용은 힘들다.

 

  • 다른 쿼리
SELECT a.*, b.*, (SELECT TOP 1 status FROM server_name.db.dbo.tbl WHERE id = a.id)
FROM a_tbl a
ORDER BY a.date DESC
OFFSET 0 ROWS FETCH NEXT 100 ROWS ONLY

15초
join이 아닌 이상 subquery로는 1개의 컬럼만 사용할 수 있다.

 

어떤 쿼리든 execution plan을 보면 view의 테이블 중 백만개의 record가 들어가있는 테이블의 clustered index scan이 cost의 상당수를 잡아먹는다
답이 안나온다.
